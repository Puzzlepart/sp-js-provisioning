# Specs

Specs describe the intended behavior and implementation plan for a change.

Use a spec when work changes product behavior, provisioning semantics, public schema, deployment behavior, or shared architecture.

Recommended lifecycle:

1. `Draft` - problem and acceptance criteria are being shaped.
2. `Ready` - implementation can start.
3. `Implemented` - code is complete and local checks pass.
4. `Verified` - behavior has been tested in the target environment.
5. `Superseded` - replaced by a newer spec.

Start from `../templates/spec.md`.
